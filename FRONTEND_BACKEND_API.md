# Переход фронтенда на запросы к бэкенду (Laravel API)

Краткое описание того, как React‑приложение ходит в API, как устроены токены и почему **сессия в `sessionStorage` и профиль с `/me` не совпадают по времени**.

---

## 1. Настройка

1. Скопируйте `.env.example` в `.env` в корне фронта (рядом с Vite) или задайте переменные в окружении.
2. **`VITE_API_BASE_URL`** — базовый URL API **без** завершающего слэша, с префиксом `/api`, как в Laravel:

   ```env
   VITE_API_BASE_URL=http://localhost:8080/api
   ```

3. Если переменная не задана, клиент по умолчанию использует `http://localhost:8080/api` (см. `src/api/client.js`).
4. Прокси в `vite.config.js` нет: браузер ходит на бэкенд напрямую. На бэке должны быть корректные **CORS**‑заголовки для origin dev‑сервера (например `http://localhost:5173`).

---

## 2. Слои кода

| Слой | Файл | Роль |
|------|------|------|
| HTTP | `src/api/http.js` | `fetch`, JSON, заголовки `Accept` / `Content-Type`, `Authorization: Bearer …`, класс **`HttpError`** при `!res.ok`. |
| Клиент API | `src/api/client.js` | Сборка путей, query, выбор токена владельца/детейлинга, методы домена (`listCars`, `loginOwner`, …). |
| Точка входа | `src/api/index.js` | `createApiClient()` / `getApi()` — один синглтон на приложение. |
| React | `src/ui/useRepo.js` | `useRepo()` возвращает API + `_version`; после мутаций вызывайте **`invalidateRepo()`**, чтобы подписчики перечитали данные. |

Новые вызовы бэка добавляются в **`createApiClient()`** в `client.js`, затем используются из страниц через **`useRepo()`** или напрямую через **`getApi()`** (например в `useDetailing.js`).

---

## 3. Авторизация

- Защищённые маршруты Laravel: **`auth:sanctum`** + для владельца **`ensure.owner`**, для детейлинга **`ensure.detailing`** (см. `backend/routes/api.php`).
- Фронт передаёт **`Authorization: Bearer <token>`** — токен читается из `sessionStorage` через `getOwnerToken()` / `getDetailingToken()` в `src/ui/auth.js`.
- Логин (`owners/login`, `detailings/login`): при ответе **401/404** с JSON вида `{ ok, reason }` клиент **не всегда бросает исключение** — см. **`authLoginReq`** в `client.js`, чтобы UI мог показать «неверный пароль», а не «ошибка сети».

---

## 4. Два «режима» API (владелец и детейлинг)

Один и тот же репозиторий на фронте ветвится по **какой токен есть**:

- С токеном владельца ходим на префикс **`/owners/...`** (машины, события, доки, шары, заявки и т.д.).
- С токеном детейлинга — на **`/cars`**, **`/events/{id}`**, **`/me`** (профиль партнёра) и связанные пути.

Это сделано в `client.js` (например `listCars`, `getCar`, `updateCar`). При добавлении эндпоинта важно не забыть обе ветки, если сценарий общий.

---

## 5. Сессия vs профиль (важно для UI)

- **`hasOwnerSession()` / `hasDetailingSession()`** (`auth.js`) — **синхронно** по данным в storage (токен + идентификаторы/почта). Ими удобно решать **редиректы** и `mode` в `DetailingSessionProvider`.
- **`owner` / `detailing`** в контексте (`useDetailing.js`) приходят **асинхронно** после **`getMeOwner()`** / **`getMeDetailing()`**.

Пока запрос `/me` не завершился, `owner` может быть **`null`**, хотя пользователь уже «залогинен». Страницы не должны обращаться к полям профиля как к `owner.field` без **`owner?.field`** или без явного состояния загрузки.

Для принудительного обновления профиля после **`PATCH .../me`** (токен тот же) используется **`bumpSessionRefresh()`** в `auth.js` — отдельно от **`invalidateRepo()`**.

---

## 6. Дедупликация запросов профиля

В `client.js` для **`getMeOwner`** и **`getMeDetailing`** есть простая защита от параллельных дублей (`inflightOwnerMe` / `inflightDetailingMe`).

---

## 7. Добавление нового метода API на фронте

1. Реализовать маршрут на бэке в `backend/routes/api.php` и контроллер.
2. В `src/api/client.js` внутри `createApiClient()` добавить async‑метод: собрать путь, при необходимости выбрать `oTok()` / `dTok()`, вызвать `req(...)`.
3. В UI вызвать через `const r = useRepo(); await r.newMethod(...)`.
4. После изменений данных, от которых зависят списки/кэш в компонентах, вызвать **`invalidateRepo()`**; после смены полей профиля в `/me` — при необходимости **`bumpSessionRefresh()`**.

---

## 8. Проверка связки

- Здоровье API: **`GET /api/health`** → `{ ok: true }`.
- Убедитесь, что **`VITE_API_BASE_URL`** указывает на тот же хост/порт, где реально слушает Laravel (в Docker часто отличается от `localhost` хоста).
