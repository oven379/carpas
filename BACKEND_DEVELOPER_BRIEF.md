# КарПас — бриф для бэкенд-разработчика

Документ описывает расхождения между фронтендом (React), заглушкой `VITE_API_MODE=real` в `src/api/index.js` и текущим Laravel API (`backend/routes/api.php`). Его можно переслать как есть.

**Дата актуализации:** 2026-04-03

---

## 1. Критично: режим `VITE_API_MODE=real` на фронте

### 1.1. Асинхронность vs синхронные вызовы в UI

В `src/api/index.js` методы `createRealApi()` объявлены как **`async`** и возвращают **Promise**. Страницы вызывают их **без `await`** (например `HomePage`: `r.listCars()`, партнёрский вход: `r.loginDetailing(...)`).

**Итог:** при включении `real` вместо данных в компоненты попадает Promise → ломаются `.length`, `.filter`, проверки `res?.ok` и т.д. **Текущий UI с `createRealApi` без доработки не согласован.**

### 1.2. Токен авторизации не передаётся

В `createRealApi` задано `const token = null`. Ответ логина/регистрации с токеном **нигде не сохраняется и не подставляется** в `httpJson`.

**Итог:** защищённые маршруты Laravel (`auth:sanctum`) **не заработают** без сохранения токена (например в памяти/`sessionStorage`) и заголовка `Authorization: Bearer <token>`.

### 1.3. Методы есть в mock, отсутствуют в `createRealApi`

В mock (`src/lib/repo.js` + обёртка в `createMockApi`) есть, в объекте **real** — **нет** (вызов даст ошибку):

- владелец: `listOwners`, `getOwner`, `registerOwner`, `loginOwner`, `updateOwner`
- заявки (VIN / привязка): `findCarsByVin`, `findCarsByPlate`, `listClaimsForOwner`, `listClaimsForDetailing`, `createClaim`, `reviewClaim`

Сценарии «Гараж», заявки, модерация у партнёра в режиме **real** этим слоем не покрыты.

### 1.4. Разные сигнатуры mock vs real stub

| Метод (концептуально) | Mock (`repo`) | Заглушка `createRealApi` |
|----------------------|---------------|---------------------------|
| `listCars` | аргумент: строка `detailingId` или объект `{ ownerEmail }` или без аргумента (все авто в демо) | без аргументов — только список «по текущему пользователю API» |
| `getCar` | `(id, scope?)` | `(id)` |
| `createCar` | `(detailingId \| scope, input)` | `(input)` |
| `listEvents`, документы и т.д. | второй аргумент — scope (владелец vs сервис) | в основном только `carId` |

Даже после перевода UI на `async`, **роль и область видимости** в mock задаются аргументами; в real их нужно либо выводить из токена на бэке, либо явно заложить в контракт API.

## 2. Текущий Laravel API vs фронт и vs mock-логика

### 2.1. Только детейлинг под Sanctum, нет владельца

Защищённые маршруты завязаны на **пользователя-детейлинга**. Отдельных эндпоинтов для **владельца**, **личного гаража** и **заявок (claims)** в `api.php` нет. В mock эти сценарии реализованы в `repo.js` и **не имеют** бэкенд-аналога.

### 2.2. Регистрация партнёра: расхождение полей и ответа

- **Mock** `registerDetailing`: обязательны имя, email, **телефон** (`bad_phone`); пароль в UI на заявке может не передаваться — в репозитории подставляется демо.
- **Laravel** `DetailingAuthController::register`: **`password` — required**; поле **`phone` в валидации отсутствует**.
- **UI** `PartnerApplyPage` при успехе делает `setSessionDetailingId(d.id)` — mock возвращает объект детейлинга `d`. **Laravel** возвращает `{ detailing: {...}, token }` — нужен **`detailing.id`**, а не `d.id` у корня ответа.

**Итог:** без согласования полей и формы ответа регистрация на бэке и сессия на фронте **разъедутся**.

### 2.3. Маршруты из заглушки фронта, которых нет в Laravel

В `createRealApi` фигурируют, в **`backend/routes/api.php` отсутствуют** (или отличают смысл):

- `GET /detailings`, `GET /detailings/{id}`
- `PATCH /detailings/{id}` — нужен для сохранения профиля детейлинга и `profileCompleted` (экран настроек вызывает `updateDetailing` с проверкой `id === detailingId` в mock).

### 2.4. События (история авто)

- **Mock:** визиты с `source === 'service'` **удалить нельзя** (`deleteEvent` возвращает `null`); есть **обновление** визита (`updateEvent` / PATCH по смыслу).
- **Laravel:** реализован **`DELETE /events/{id}`** для событий детейлинга; **PATCH для события нет**. Инвариант «сервисная запись не удаляется» на бэке **не совпадает** с mock.
- Поля **`source`, `ownerEmail`, `maintenanceServices`** и разделение «история сервиса / личная история владельца» в mock есть; в показанном `CarEventController` — **упрощённая сервисная модель** без этих осей.

### 2.5. Публичная ссылка

`GET /share/{token}` в Laravel есть — ближе всего к `getCarByShareToken`. Нужно сверить **форму JSON** с ожиданиями `PublicCarPage` (фильтрация событий/документов по `source === 'owner'` и т.д.).

## 3. Замечания только по mock (`repo.js`) — на будущее при переносе логики на бэк

1. **`deleteCar`** не удаляет записи из хранилища заявок (`CLAIMS_KEY`) — возможны «висячие» заявки на удалённый `carId`.
2. **`listDetailings`**: если в storage окажется не массив, возможен падение на `.map` (защита массивом — пожелание).
3. **`reviewClaim` (approved)** повторно может перезаписать владельца на машине — явного инварианта «один подтверждённый владелец на авто» нет.
4. Внутри репозитория местами используется **`getCar(carId)` без scope** — при глобально уникальных id это ок.

## 4. Сводная таблица сценариев

| Сценарий | Mock | Real + текущий Laravel |
|----------|------|-------------------------|
| Главная без входа | `listCars()` без scope | Promise + нет совпадения контракта |
| Владелец: вход/гараж | локальная сессия + `listCars({ ownerEmail })` | нет API |
| Заявка владельца по VIN | `findCarsByVin`, `createClaim` | нет API и методов в real |
| Партнёр: очередь заявок | `listClaimsForDetailing`, `reviewClaim` | нет API |
| Партнёр: список авто | фильтр по `detailingId` + approved claim для машин с `ownerEmail` | Laravel: все авто детейлинга — **другая модель доступа** |
| Партнёр: настройки профиля | `updateDetailing` | нет `PATCH` детейлинга в маршрутах |
| История у владельца | смешанный фильтр owner + service от привязанного детейлинга | API заточен под детейлинг |

## 5. Чек-лист для бэкенда (кратко)

1. Договориться о **Bearer-токене** после логина/регистрации и едином способе передачи с фронта.
2. Согласовать **регистрацию партнёра**: обязательные поля (пароль, телефон), форма ответа `{ detailing, token }` и правки UI при необходимости.
3. Реализовать **обновление профиля детейлинга** (в т.ч. флаг завершения онбординга), если он нужен продукту.
4. Спроектировать **модель владельца + заявок (claims)** и REST (или RPC) под фронтовые сценарии.
5. События: правила **удаления/редактирования** согласовать с продуктом; при необходимости **PATCH** и поля `source` / права владельца vs сервиса.
6. На фронте при включении real — **слой async** (не вызывать API как синхронный `repo`).
